# ТЗ: сервис синхронизации Spotify Likes → Soundeo

## 1. Цель

Разработать локальный или серверный сервис-агент, который:

* раз в день получает список треков, лайкнутых в Spotify;
* ищет эти треки на `soundeo.com` под моей учётной записью;
* если трек найден и доступен — помечает его звездой;
* если трек найден, но пока недоступен для нужного действия — ставит лайк;
* если трек не найден совсем — добавляет запись в waitlist;
* по возможности проверяет, не был ли трек уже скачан ранее, и не делает повторных действий;
* при первичном запуске выполняет полную сверку:

  * берёт все лайкнутые треки из Spotify,
  * сверяет с уже скачанными треками на Soundeo,
  * если трек есть в Spotify likes, но отсутствует в списке скачанных на Soundeo — обрабатывает его по основным правилам.

---

## 2. Формат работы

### 2.1 Основной режим

* Периодичность проверки Spotify likes: **1 раз в сутки**.
* Обрабатываются:

  * новые лайки с момента последнего запуска;
  * опционально — повторная проверка ранее не найденных / отложенных треков.

### 2.2 Первичная синхронизация

При первом запуске:

* получить весь список Spotify Liked Tracks;
* получить историю/список скачанных треков на Soundeo;
* построить сопоставление;
* для каждого трека из Spotify:

  * если уже скачан на Soundeo — пометить как `downloaded_already`, действий не делать;
  * если не скачан — искать на Soundeo и выполнять правила обработки.

---

## 3. Бизнес-логика

Для каждого трека из Spotify:

### 3.1 Нормализация данных

Перед поиском привести трек к каноническому виду:

* artist(s)
* title
* remix/version/mix
* duration
* ISRC, если доступен
* release name, если доступен

Нужно уметь чистить:

* `Original Mix`
* `Extended Mix`
* скобки, feat./ft.
* дубль-пробелы
* регистр
* спецсимволы

### 3.2 Поиск на Soundeo

Искать по нескольким стратегиям:

1. `artist + title`
2. `artist + title + remix/version`
3. альтернативные варианты нормализованной строки

### 3.3 Правила действий

После поиска на Soundeo:

**Сценарий A: трек найден и доступен**

* если не скачан ранее:

  * отметить звездой;
  * записать статус `starred`;
* если уже скачан:

  * ничего не делать;
  * записать статус `downloaded_already`.

**Сценарий B: трек найден, но недоступен для скачивания / ещё не вышел / нет релиза в нужном виде**

* поставить лайк;
* записать статус `liked_waiting_availability`.

**Сценарий C: трек не найден**

* добавить в локальный waitlist;
* записать статус `not_found_waitlist`.

### 3.4 Повторная обработка

Для waitlist:

* раз в день пытаться повторно искать треки со статусами:

  * `liked_waiting_availability`
  * `not_found_waitlist`

## 4. Источники данных

### 4.1 Spotify

Источник истины по лайкам: Spotify Saved Tracks / Liked Songs. Spotify Web API поддерживает получение сохранённых треков пользователя и возвращает `added_at`, а для доступа нужен scope `user-library-read`. ([developer.spotify.com][1])

Использовать:

* OAuth 2.0
* endpoint получения saved tracks
* пагинацию
* хранение last_sync_cursor / last_sync_time

### 4.2 Soundeo

Так как управление на сайте выполняется хоткеями, основной вариант интеграции:

* **browser automation** через Playwright;
* вход под моей учётной записью;
* работа либо через DOM-элементы, либо через эмуляцию клавиш;
* предпочтительно сначала пробовать DOM/selectors, а hotkeys использовать как fallback.

---

## 5. Техническая архитектура

### 5.1 Компоненты

1. **Spotify client**

   * OAuth
   * чтение liked tracks
   * пагинация
   * инкрементальная синхронизация

2. **Normalizer / Matcher**

   * нормализация названий
   * fuzzy matching
   * дедупликация

3. **Soundeo automation worker**

   * логин
   * поиск треков
   * открытие карточки / строки результата
   * определение статуса трека
   * star / like / skip
   * чтение раздела downloaded

4. **Local state storage**

   * SQLite/Postgres
   * таблицы треков, действий, waitlist, downloaded_cache

5. **Scheduler**

   * cron / systemd timer / GitHub Actions / Docker scheduled job
   * ежедневный запуск

6. **Logging/Reporting**

   * логи
   * CSV/JSON отчёт
   * итог по каждому запуску

### 5.2 Рекомендуемый стек

* Python 3.12+
* Playwright
* SQLite
* spotipy или собственный thin client для Spotify API
* rapidfuzz для матчингa
* pydantic для конфигов
* Docker — опционально

## 7. Логика “уже скачан ли трек”

Проверка “не скачан ли уже трек” должна работать в 3 слоя:

1. **Прямое наличие в локальной БД**
   Если ранее уже зафиксировано, что этот трек скачан — повторно не обрабатывать.

2. **Периодический парсинг раздела Downloaded на Soundeo**
   Агент должен уметь открыть раздел скачанных и построить локальный кэш.

3. **Fuzzy match при расхождении названий**
   Если у Spotify и Soundeo немного разные строки, использовать правила сопоставления:

   * совпадение по artist + title,
   * бонус за remix/version,
   * бонус за duration,
   * бонус за ISRC при наличии.

---

## 8. Требования к надёжности

* Не выполнять одно и то же действие дважды для одного трека без необходимости.
* Уметь продолжать после падения.
* Все действия должны быть идемпотентными насколько возможно.
* Хранить скриншоты и HTML страницы при ошибке.
* Ограничить частоту действий, чтобы не выглядеть как агрессивный бот.
* Поддерживать ручной dry-run режим без реальных кликов.

---

## 9. Режимы запуска

### 9.1 Dry run

* ничего не нажимать на Soundeo;
* только искать, определять статус и формировать отчёт.

### 9.2 Active sync

* выполнять реальные действия:

  * star
  * like
  * waitlist update

### 9.3 Initial full sync

* полная сверка всех Spotify likes и Soundeo downloaded.

---

## 10. Отчёт по запуску

После каждого запуска формировать отчёт:

* всего лайков в Spotify обработано;
* новых лайков найдено;
* уже скачанных;
* помечено звездой;
* поставлено лайков;
* добавлено в waitlist;
* ошибок;
* список проблемных треков.

Формат:

* консольный summary,
* JSON файл,
* опционально Telegram/email уведомление.

---

## 11. Ограничения и риски

* Если у Soundeo нет публичного API, интеграция будет держаться на браузерной автоматизации.
* Любое изменение верстки/хоткеев на сайте может ломать сценарий.
* Нужно учитывать anti-bot механики, капчу, нестабильные селекторы, lazy loading.
* Следует отдельно проверить, допустимо ли это с точки зрения правил сервиса.
* Для надёжности лучше опираться не только на хоткеи, а сначала пытаться работать по DOM-элементам, если они стабильны.

# Как я бы сформулировал задачу для Codex / Claude Code

Ниже уже почти готовый промпт для агента.

## Промпт агенту

Разработай Python-сервис `spotify-soundeo-sync`, который синхронизирует мои Spotify Liked Songs с сайтом soundeo.com.

Функциональные требования:

* получать мои liked tracks из Spotify через Web API;
* хранить локальное состояние в SQLite;
* при первичной синхронизации загрузить все liked tracks и сверить их с уже скачанными треками на Soundeo;
* если трек уже скачан — ничего не делать и помечать как `downloaded_already`;
* если трек найден на Soundeo и доступен — помечать звездой;
* если трек найден, но ещё недоступен для нужного действия — ставить лайк;
* если трек вообще не найден — добавлять в локальный waitlist;
* раз в день обрабатывать новые лайки и повторно проверять waitlist.

Технические требования:

* стек: Python 3.12+, Playwright, SQLite, pydantic;
* поддержать режимы `dry-run`, `initial-sync`, `daily-sync`;
* использовать browser automation для Soundeo;
* сначала пытаться работать через стабильные DOM selectors, hotkeys использовать как fallback;
* реализовать нормализацию строк и fuzzy matching для сопоставления треков;
* не выполнять повторные действия, если трек уже обработан;
* вести детальные логи, делать скриншоты и HTML dump при ошибках;
* оформить проект как production-ready repository с README, `.env.example`, миграциями БД, тестами для normalizer/matcher и CLI-командами.

CLI:

* `python -m app initial-sync`
* `python -m app daily-sync`
* `python -m app retry-waitlist`
* `python -m app dry-run`

Структура проекта:

* `/src`
* `/data`
* `/logs`
* `/artifacts/screenshots`
* `/artifacts/html`

Также:

* подготовь README с инструкцией по настройке Spotify OAuth;
* добавь cron/systemd пример запуска раз в сутки;
* заложи конфигurable retry policy и rate limiting;
* все секреты брать только из `.env`.
